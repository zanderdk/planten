42 <a name="intro"></a>
43 Intro
44 =====
45 
46  This project is split in two. The library and the client. The client part
47  uses the library, but the library is designed to allow other applications to
48  use it.
49 
50  The largest amount of code and complexity is in the library part.

53 <a name="git"></a>
54 git
55 ===
56 
57  All changes to the sources are committed to the git repository as soon as
58  they're somewhat verified to work. Changes shall be committed as independently
59  as possible so that individual changes can be easily spotted and tracked
60  afterwards.
61 
62  Tagging shall be used extensively, and by the time we release new archives we
63  should tag the sources with a name similar to the released version number.

124 <a name="winvsunix"></a>
125 Windows vs Unix
126 ===============
127 
128  There are a few differences in how to program curl the Unix way compared to
129  the Windows way. Perhaps the four most notable details are:
130 
131  1. Different function names for socket operations.
132 
133  In curl, this is solved with defines and macros, so that the source looks
134  the same in all places except for the header file that defines them. The
135  macros in use are sclose(), sread() and swrite().
136 
137  2. Windows requires a couple of init calls for the socket stuff.
138 
139  That's taken care of by the `curl_global_init()` call, but if other libs
140  also do it etc there might be reasons for applications to alter that
141  behaviour.
142 
143  3. The file descriptors for network communication and file operations are
144  not as easily interchangeable as in Unix.
145 
146  We avoid this by not trying any funny tricks on file descriptors.
147 
148  4. When writing data to stdout, Windows makes end-of-lines the DOS way, thus
149  destroying binary data, although you do want that conversion if it is
150  text coming through... (sigh)
151 
152  We set stdout to binary under windows
153 
154  Inside the source code, We make an effort to avoid `#ifdef [Your OS]`. All
155  conditionals that deal with features *should* instead be in the format
156  `#ifdef HAVE_THAT_WEIRD_FUNCTION`. Since Windows can't run configure scripts,
157  we maintain a `curl_config-win32.h` file in lib directory that is supposed to
158  look exactly like a `curl_config.h` file would have looked like on a Windows
159  machine!
160 
161  Generally speaking: always remember that this will be compiled on dozens of
162  operating systems. Don't walk on the edge!

164 <a name="Library"></a>
165 Library
166 =======
167 
168  (See [Structs in libcurl](#structs) for the separate section describing all
169  major internal structs and their purposes.)
170 
171  There are plenty of entry points to the library, namely each publicly defined
172  function that libcurl offers to applications. All of those functions are
173  rather small and easy-to-follow. All the ones prefixed with `curl_easy` are
174  put in the lib/easy.c file.
175 
176  `curl_global_init()` and `curl_global_cleanup()` should be called by the
177  application to initialize and clean up global stuff in the library. As of
178  today, it can handle the global SSL initing if SSL is enabled and it can init
179  the socket layer on windows machines. libcurl itself has no "global" scope.
180 
181  All printf()-style functions use the supplied clones in lib/mprintf.c. This
182  makes sure we stay absolutely platform independent.
183 
184  [ `curl_easy_init()`][2] allocates an internal struct and makes some
185  initializations. The returned handle does not reveal internals. This is the
186  `Curl_easy` struct which works as an "anchor" struct for all `curl_easy`
187  functions. All connections performed will get connect-specific data allocated
188  that should be used for things related to particular connections/requests.
189 
190  [`curl_easy_setopt()`][1] takes three arguments, where the option stuff must
191  be passed in pairs: the parameter-ID and the parameter-value. The list of
192  options is documented in the man page. This function mainly sets things in
193  the `Curl_easy` struct.
194 
195  `curl_easy_perform()` is just a wrapper function that makes use of the multi
196  API. It basically calls `curl_multi_init()`, `curl_multi_add_handle()`,
197  `curl_multi_wait()`, and `curl_multi_perform()` until the transfer is done
198  and then returns.
199 
200  Some of the most important key functions in url.c are called from multi.c
201  when certain key steps are to be made in the transfer operation.
202 
203 <a name="Curl_connect"></a>
204 Curl_connect()
205 --------------
206 
207  Analyzes the URL, it separates the different components and connects to the
208  remote host. This may involve using a proxy and/or using SSL. The
209  `Curl_resolv()` function in lib/hostip.c is used for looking up host names
210  (it does then use the proper underlying method, which may vary between
211  platforms and builds).
212 
213  When `Curl_connect` is done, we are connected to the remote site. Then it
214  is time to tell the server to get a document/file. `Curl_do()` arranges
215  this.
216 
217  This function makes sure there's an allocated and initiated 'connectdata'
218  struct that is used for this particular connection only (although there may
219  be several requests performed on the same connect). A bunch of things are
220  inited/inherited from the `Curl_easy` struct.
221 
222 <a name="Curl_do"></a>
223 Curl_do()
224 ---------
225 
226  `Curl_do()` makes sure the proper protocol-specific function is called. The
227  functions are named after the protocols they handle.
228 
229  The protocol-specific functions of course deal with protocol-specific
230  negotiations and setup. They have access to the `Curl_sendf()` (from
231  lib/sendf.c) function to send printf-style formatted data to the remote
232  host and when they're ready to make the actual file transfer they call the
233  `Curl_Transfer()` function (in lib/transfer.c) to setup the transfer and
234  returns.
235 
236  If this DO function fails and the connection is being re-used, libcurl will
237  then close this connection, setup a new connection and re-issue the DO
238  request on that. This is because there is no way to be perfectly sure that
239  we have discovered a dead connection before the DO function and thus we
240  might wrongly be re-using a connection that was closed by the remote peer.
241 
242  Some time during the DO function, the `Curl_setup_transfer()` function must
243  be called with some basic info about the upcoming transfer: what socket(s)
244  to read/write and the expected file transfer sizes (if known).
245 
246 <a name="Curl_readwrite"></a>
247 Curl_readwrite()
248 ----------------
249 
250  Called during the transfer of the actual protocol payload.
251 
252  During transfer, the progress functions in lib/progress.c are called at
253  frequent intervals (or at the user's choice, a specified callback might get
254  called). The speedcheck functions in lib/speedcheck.c are also used to
255  verify that the transfer is as fast as required.
256 
257 <a name="Curl_done"></a>
258 Curl_done()
259 -----------
260 
261  Called after a transfer is done. This function takes care of everything
262  that has to be done after a transfer. This function attempts to leave
263  matters in a state so that `Curl_do()` should be possible to call again on
264  the same connection (in a persistent connection case). It might also soon
265  be closed with `Curl_disconnect()`.
266 
267 <a name="Curl_disconnect"></a>
268 Curl_disconnect()
269 -----------------
270 
271  When doing normal connections and transfers, no one ever tries to close any
272  connections so this is not normally called when `curl_easy_perform()` is
273  used. This function is only used when we are certain that no more transfers
274  are going to be made on the connection. It can be also closed by force, or
275  it can be called to make sure that libcurl doesn't keep too many
276  connections alive at the same time.
277 
278  This function cleans up all resources that are associated with a single
279  connection.

281 <a name="http"></a>
282 HTTP(S)
283 =======
284 
285  HTTP offers a lot and is the protocol in curl that uses the most lines of
286  code. There is a special file (lib/formdata.c) that offers all the multipart
287  post functions.
288 
289  base64-functions for user+password stuff (and more) is in (lib/base64.c) and
290  all functions for parsing and sending cookies are found in (lib/cookie.c).
291 
292  HTTPS uses in almost every case the same procedure as HTTP, with only two
293  exceptions: the connect procedure is different and the function used to read
294  or write from the socket is different, although the latter fact is hidden in
295  the source by the use of `Curl_read()` for reading and `Curl_write()` for
296  writing data to the remote server.
297 
298  `http_chunks.c` contains functions that understands HTTP 1.1 chunked transfer
299  encoding.
300 
301  An interesting detail with the HTTP(S) request, is the `Curl_add_buffer()`
302  series of functions we use. They append data to one single buffer, and when
303  the building is finished the entire request is sent off in one single write.
304  This is done this way to overcome problems with flawed firewalls and lame
305  servers.

381 <a name="persistent"></a>
382 Persistent Connections
383 ======================
384 
385  The persistent connection support in libcurl requires some considerations on
386  how to do things inside of the library.
387 
388  - The `Curl_easy` struct returned in the [`curl_easy_init()`][2] call
389  must never hold connection-oriented data. It is meant to hold the root data
390  as well as all the options etc that the library-user may choose.
391 
392  - The `Curl_easy` struct holds the "connection cache" (an array of
393  pointers to 'connectdata' structs).
394 
395  - This enables the 'curl handle' to be reused on subsequent transfers.
396 
397  - When libcurl is told to perform a transfer, it first checks for an already
398  existing connection in the cache that we can use. Otherwise it creates a
399  new one and adds that to the cache. If the cache is full already when a new
400  connection is added, it will first close the oldest unused one.
401 
402  - When the transfer operation is complete, the connection is left
403  open. Particular options may tell libcurl not to, and protocols may signal
404  closure on connections and then they won't be kept open, of course.
405 
406  - When `curl_easy_cleanup()` is called, we close all still opened connections,
407  unless of course the multi interface "owns" the connections.
408 
409  The curl handle must be re-used in order for the persistent connections to
410  work.
411 
412 <a name="multi"></a>
413 multi interface/non-blocking
414 ============================
415 
416  The multi interface is a non-blocking interface to the library. To make that
417  interface work as well as possible, no low-level functions within libcurl
418  must be written to work in a blocking manner. (There are still a few spots
419  violating this rule.)
420 
421  One of the primary reasons we introduced c-ares support was to allow the name
422  resolve phase to be perfectly non-blocking as well.
423 
424  The FTP and the SFTP/SCP protocols are examples of how we adapt and adjust
425  the code to allow non-blocking operations even on multi-stage command-
426  response protocols. They are built around state machines that return when
427  they would otherwise block waiting for data. The DICT, LDAP and TELNET
428  protocols are crappy examples and they are subject for rewrite in the future
429  to better fit the libcurl protocol family.
430 
431 <a name="ssl"></a>
432 SSL libraries
433 =============
434 
435  Originally libcurl supported SSLeay for SSL/TLS transports, but that was then
436  extended to its successor OpenSSL but has since also been extended to several
437  other SSL/TLS libraries and we expect and hope to further extend the support
438  in future libcurl versions.
439 
440  To deal with this internally in the best way possible, we have a generic SSL
441  function API as provided by the vtls/vtls.[ch] system, and they are the only
442  SSL functions we must use from within libcurl. vtls is then crafted to use
443  the appropriate lower-level function calls to whatever SSL library that is in
444  use. For example vtls/openssl.[ch] for the OpenSSL library.

446 <a name="symbols"></a>
447 Library Symbols
448 ===============
449 
450  All symbols used internally in libcurl must use a `Curl_` prefix if they're
451  used in more than a single file. Single-file symbols must be made static.
452  Public ("exported") symbols must use a `curl_` prefix. (There are exceptions,
453  but they are to be changed to follow this pattern in future versions.) Public
454  API functions are marked with `CURL_EXTERN` in the public header files so
455  that all others can be hidden on platforms where this is possible.
456 
457 <a name="returncodes"></a>
458 Return Codes and Informationals
459 ===============================
460 
461  I've made things simple. Almost every function in libcurl returns a CURLcode,
462  that must be `CURLE_OK` if everything is OK or otherwise a suitable error
463  code as the curl/curl.h include file defines. The very spot that detects an
464  error must use the `Curl_failf()` function to set the human-readable error
465  description.
466 
467  In aiding the user to understand what's happening and to debug curl usage, we
468  must supply a fair number of informational messages by using the
469  `Curl_infof()` function. Those messages are only displayed when the user
470  explicitly asks for them. They are best used when revealing information that
471  isn't otherwise obvious.

481 <a name="client"></a>
482 Client
483 ======
484 
485  main() resides in `src/tool_main.c`.
486 
487  `src/tool_hugehelp.c` is automatically generated by the mkhelp.pl perl script
488  to display the complete "manual" and the `src/tool_urlglob.c` file holds the
489  functions used for the URL-"globbing" support. Globbing in the sense that the
490  {} and [] expansion stuff is there.
491 
492  The client mostly sets up its 'config' struct properly, then
493  it calls the `curl_easy_*()` functions of the library and when it gets back
494  control after the `curl_easy_perform()` it cleans up the library, checks
495  status and exits.
496 
497  When the operation is done, the ourWriteOut() function in src/writeout.c may
498  be called to report about the operation. That function is using the
499  `curl_easy_getinfo()` function to extract useful information from the curl
500  session.
501 
502  It may loop and do all this several times if many URLs were specified on the
503  command line or config file.

505 <a name="memorydebug"></a>
506 Memory Debugging
507 ================
508 
509  The file lib/memdebug.c contains debug-versions of a few functions. Functions
510  such as malloc, free, fopen, fclose, etc that somehow deal with resources
511  that might give us problems if we "leak" them. The functions in the memdebug
512  system do nothing fancy, they do their normal function and then log
513  information about what they just did. The logged data can then be analyzed
514  after a complete session,
515 
516  memanalyze.pl is the perl script present in tests/ that analyzes a log file
517  generated by the memory tracking system. It detects if resources are
518  allocated but never freed and other kinds of errors related to resource
519  management.
520 
521  Internally, definition of preprocessor symbol DEBUGBUILD restricts code which
522  is only compiled for debug enabled builds. And symbol CURLDEBUG is used to
523  differentiate code which is _only_ used for memory tracking/debugging.
524 
525  Use -DCURLDEBUG when compiling to enable memory debugging, this is also
526  switched on by running configure with --enable-curldebug. Use -DDEBUGBUILD
527  when compiling to enable a debug build or run configure with --enable-debug.
528 
529  curl --version will list 'Debug' feature for debug enabled builds, and
530  will list 'TrackMemory' feature for curl debug memory tracking capable
531  builds. These features are independent and can be controlled when running
532  the configure script. When --enable-debug is given both features will be
533  enabled, unless some restriction prevents memory tracking from being used.

597 curlx
598 =====
599 
600  The libcurl source code offers a few functions by source only. They are not
601  part of the official libcurl API, but the source files might be useful for
602  others so apps can optionally compile/build with these sources to gain
603  additional functions.
604 
605  We provide them through a single header file for easy access for apps:
606  "curlx.h"
607 
608 `curlx_strtoofft()`
609 -------------------
610  A macro that converts a string containing a number to a `curl_off_t` number.
611  This might use the `curlx_strtoll()` function which is provided as source
612  code in strtoofft.c. Note that the function is only provided if no
613  strtoll() (or equivalent) function exist on your platform. If `curl_off_t`
614  is only a 32 bit number on your platform, this macro uses strtol().
615

625  - `curlx_getenv`

639 <a name="contentencoding"></a>
640 Content Encoding
641 ================
642 
643 ## About content encodings
644 
645  [HTTP/1.1][4] specifies that a client may request that a server encode its
646  response. This is usually used to compress a response using one (or more)
647  encodings from a set of commonly available compression techniques. These
648  schemes include 'deflate' (the zlib algorithm), 'gzip' 'br' (brotli) and
649  'compress'. A client requests that the server perform an encoding by including
650  an Accept-Encoding header in the request document. The value of the header
651  should be one of the recognized tokens 'deflate', ... (there's a way to
652  register new schemes/tokens, see sec 3.5 of the spec). A server MAY honor
653  the client's encoding request. When a response is encoded, the server
654  includes a Content-Encoding header in the response. The value of the
655  Content-Encoding header indicates which encodings were used to encode the
656  data, in the order in which they were applied.
657 
658  It's also possible for a client to attach priorities to different schemes so
659  that the server knows which it prefers. See sec 14.3 of RFC 2616 for more
660  information on the Accept-Encoding header. See sec [3.1.2.2 of RFC 7231][15]
661  for more information on the Content-Encoding header.
662 
663 ## Supported content encodings
664 
665  The 'deflate', 'gzip' and 'br' content encodings are supported by libcurl.
666  Both regular and chunked transfers work fine. The zlib library is required
667  for the 'deflate' and 'gzip' encodings, while the brotli decoding library is
668  for the 'br' encoding.
669 
670 ## The libcurl interface
671 
672  To cause libcurl to request a content encoding use:
673 
674  [`curl_easy_setopt`][1](curl, [`CURLOPT_ACCEPT_ENCODING`][5], string)
675 
676  where string is the intended value of the Accept-Encoding header.
677 
678  Currently, libcurl does support multiple encodings but only
679  understands how to process responses that use the "deflate", "gzip" and/or
680  "br" content encodings, so the only values for [`CURLOPT_ACCEPT_ENCODING`][5]
681  that will work (besides "identity," which does nothing) are "deflate",
682  "gzip" and "br". If a response is encoded using the "compress" or methods,
683  libcurl will return an error indicating that the response could
684  not be decoded. If `<string>` is NULL no Accept-Encoding header is generated.
685  If `<string>` is a zero-length string, then an Accept-Encoding header
686  containing all supported encodings will be generated.
687 
688  The [`CURLOPT_ACCEPT_ENCODING`][5] must be set to any non-NULL value for
689  content to be automatically decoded. If it is not set and the server still
690  sends encoded content (despite not having been asked), the data is returned
691  in its raw form and the Content-Encoding type is not checked.
692 
693 ## The curl interface
694 
695  Use the [--compressed][6] option with curl to cause it to ask servers to
696  compress responses using any format supported by curl.

698 <a name="hostip"></a>
699 hostip.c explained
700 ==================
701 
702  The main compile-time defines to keep in mind when reading the host*.c source
703  file are these:
704 
705 ## `CURLRES_IPV6`
706 
707  this host has getaddrinfo() and family, and thus we use that. The host may
708  not be able to resolve IPv6, but we don't really have to take that into
709  account. Hosts that aren't IPv6-enabled have `CURLRES_IPV4` defined.
710 
711 ## `CURLRES_ARES`
712 
713  is defined if libcurl is built to use c-ares for asynchronous name
714  resolves. This can be Windows or *nix.
715 
716 ## `CURLRES_THREADED`
717 
718  is defined if libcurl is built to use threading for asynchronous name
719  resolves. The name resolve will be done in a new thread, and the supported
720  asynch API will be the same as for ares-builds. This is the default under
721  (native) Windows.
722 
723  If any of the two previous are defined, `CURLRES_ASYNCH` is defined too. If
724  libcurl is not built to use an asynchronous resolver, `CURLRES_SYNCH` is
725  defined.
726 
727 ## host*.c sources
728 
729  The host*.c sources files are split up like this:
730 
731  - hostip.c - method-independent resolver functions and utility functions
732  - hostasyn.c - functions for asynchronous name resolves
733  - hostsyn.c - functions for synchronous name resolves
734  - asyn-ares.c - functions for asynchronous name resolves using c-ares
735  - asyn-thread.c - functions for asynchronous name resolves using threads
736  - hostip4.c - IPv4 specific functions
737  - hostip6.c - IPv6 specific functions
738 
739  The hostip.h is the single united header file for all this. It defines the
740  `CURLRES_*` defines based on the config*.h and `curl_setup.h` defines.

743 Track Down Memory Leaks
744 =======================
745 
746 ## Single-threaded
747 
748  Please note that this memory leak system is not adjusted to work in more
749  than one thread. If you want/need to use it in a multi-threaded app. Please
750  adjust accordingly.
751 
752 
753 ## Build
754 
755  Rebuild libcurl with -DCURLDEBUG (usually, rerunning configure with
756  --enable-debug fixes this). 'make clean' first, then 'make' so that all
757  files are actually rebuilt properly. It will also make sense to build
758  libcurl with the debug option (usually -g to the compiler) so that debugging
759  it will be easier if you actually do find a leak in the library.
760 
761  This will create a library that has memory debugging enabled.
762 
763 ## Modify Your Application
764 
765  Add a line in your application code:
766 
767  `curl_memdebug("dump");`
768 
769  This will make the malloc debug system output a full trace of all resource
770  using functions to the given file name. Make sure you rebuild your program
771  and that you link with the same libcurl you built for this purpose as
772  described above.
773 
774 ## Run Your Application
775 
776  Run your program as usual. Watch the specified memory trace file grow.
777 
778  Make your program exit and use the proper libcurl cleanup functions etc. So
779  that all non-leaks are returned/freed properly.
780 
781 ## Analyze the Flow
782 
783  Use the tests/memanalyze.pl perl script to analyze the dump file:
784 
785  tests/memanalyze.pl dump
786 
787  This now outputs a report on what resources that were allocated but never
788  freed etc. This report is very fine for posting to the list!
789 
790  If this doesn't produce any output, no leak was detected in libcurl. Then
791  the leak is mostly likely to be in your code.

793 <a name="multi_socket"></a>
794 `multi_socket`
795 ==============
796 
797  Implementation of the `curl_multi_socket` API
798 
799  The main ideas of this API are simply:
800 
801  1 - The application can use whatever event system it likes as it gets info
802  from libcurl about what file descriptors libcurl waits for what action
803  on. (The previous API returns `fd_sets` which is very select()-centric).
804 
805  2 - When the application discovers action on a single socket, it calls
806  libcurl and informs that there was action on this particular socket and
807  libcurl can then act on that socket/transfer only and not care about
808  any other transfers. (The previous API always had to scan through all
809  the existing transfers.)
810 
811  The idea is that [`curl_multi_socket_action()`][7] calls a given callback
812  with information about what socket to wait for what action on, and the
813  callback only gets called if the status of that socket has changed.
814 
815  We also added a timer callback that makes libcurl call the application when
816  the timeout value changes, and you set that with [`curl_multi_setopt()`][9]
817  and the [`CURLMOPT_TIMERFUNCTION`][10] option. To get this to work,
818  Internally, there's an added struct to each easy handle in which we store
819  an "expire time" (if any). The structs are then "splay sorted" so that we
820  can add and remove times from the linked list and yet somewhat swiftly
821  figure out both how long there is until the next nearest timer expires
822  and which timer (handle) we should take care of now. Of course, the upside
823  of all this is that we get a [`curl_multi_timeout()`][8] that should also
824  work with old-style applications that use [`curl_multi_perform()`][11].
825 
826  We created an internal "socket to easy handles" hash table that given
827  a socket (file descriptor) returns the easy handle that waits for action on
828  that socket. This hash is made using the already existing hash code
829  (previously only used for the DNS cache).
830 
831  To make libcurl able to report plain sockets in the socket callback, we had
832  to re-organize the internals of the [`curl_multi_fdset()`][12] etc so that
833  the conversion from sockets to `fd_sets` for that function is only done in
834  the last step before the data is returned. I also had to extend c-ares to
835  get a function that can return plain sockets, as that library too returned
836  only `fd_sets` and that is no longer good enough. The changes done to c-ares
837  are available in c-ares 1.3.1 and later.
838 
839 <a name="structs"></a>
840 Structs in libcurl
841 ==================
842 
843 This section should cover 7.32.0 pretty accurately, but will make sense even
844 for older and later versions as things don't change drastically that often.
845 
846 ## Curl_easy
847 
848  The `Curl_easy` struct is the one returned to the outside in the external API
849  as a "CURL *". This is usually known as an easy handle in API documentations
850  and examples.
851 
852  Information and state that is related to the actual connection is in the
853  'connectdata' struct. When a transfer is about to be made, libcurl will
854  either create a new connection or re-use an existing one. The particular
855  connectdata that is used by this handle is pointed out by
856  `Curl_easy->easy_conn`.
857 
858  Data and information that regard this particular single transfer is put in
859  the SingleRequest sub-struct.
860 
861  When the `Curl_easy` struct is added to a multi handle, as it must be in
862  order to do any transfer, the ->multi member will point to the `Curl_multi`
863  struct it belongs to. The ->prev and ->next members will then be used by the
864  multi code to keep a linked list of `Curl_easy` structs that are added to
865  that same multi handle. libcurl always uses multi so ->multi *will* point to
866  a `Curl_multi` when a transfer is in progress.
867 
868  ->mstate is the multi state of this particular `Curl_easy`. When
869  `multi_runsingle()` is called, it will act on this handle according to which
870  state it is in. The mstate is also what tells which sockets to return for a
871  specific `Curl_easy` when [`curl_multi_fdset()`][12] is called etc.
872 
873  The libcurl source code generally use the name 'data' for the variable that
874  points to the `Curl_easy`.
875 
876  When doing multiplexed HTTP/2 transfers, each `Curl_easy` is associated with
877  an individual stream, sharing the same connectdata struct. Multiplexing
878  makes it even more important to keep things associated with the right thing!
879 
880 ## connectdata
881 
882  A general idea in libcurl is to keep connections around in a connection
883  "cache" after they have been used in case they will be used again and then
884  re-use an existing one instead of creating a new as it creates a significant
885  performance boost.
886 
887  Each 'connectdata' identifies a single physical connection to a server. If
888  the connection can't be kept alive, the connection will be closed after use
889  and then this struct can be removed from the cache and freed.
890 
891  Thus, the same `Curl_easy` can be used multiple times and each time select
892  another connectdata struct to use for the connection. Keep this in mind, as
893  it is then important to consider if options or choices are based on the
894  connection or the `Curl_easy`.
895 
896  Functions in libcurl will assume that connectdata->data points to the
897  `Curl_easy` that uses this connection (for the moment).
898 
899  As a special complexity, some protocols supported by libcurl require a
900  special disconnect procedure that is more than just shutting down the
901  socket. It can involve sending one or more commands to the server before
902  doing so. Since connections are kept in the connection cache after use, the
903  original `Curl_easy` may no longer be around when the time comes to shut down
904  a particular connection. For this purpose, libcurl holds a special dummy
905  `closure_handle` `Curl_easy` in the `Curl_multi` struct to use when needed.
906 
907  FTP uses two TCP connections for a typical transfer but it keeps both in
908  this single struct and thus can be considered a single connection for most
909  internal concerns.
910 
911  The libcurl source code generally use the name 'conn' for the variable that
912  points to the connectdata.
913 
914 ## Curl_multi
915 
916  Internally, the easy interface is implemented as a wrapper around multi
917  interface functions. This makes everything multi interface.
918 
919  `Curl_multi` is the multi handle struct exposed as "CURLM *" in external
920  APIs.
921 
922  This struct holds a list of `Curl_easy` structs that have been added to this
923  handle with [`curl_multi_add_handle()`][13]. The start of the list is
924  `->easyp` and `->num_easy` is a counter of added `Curl_easy`s.
925 
926  `->msglist` is a linked list of messages to send back when
927  [`curl_multi_info_read()`][14] is called. Basically a node is added to that
928  list when an individual `Curl_easy`'s transfer has completed.
929 
930  `->hostcache` points to the name cache. It is a hash table for looking up
931  name to IP. The nodes have a limited life time in there and this cache is
932  meant to reduce the time for when the same name is wanted within a short
933  period of time.
934 
935  `->timetree` points to a tree of `Curl_easy`s, sorted by the remaining time
936  until it should be checked - normally some sort of timeout. Each `Curl_easy`
937  has one node in the tree.
938 
939  `->sockhash` is a hash table to allow fast lookups of socket descriptor for
940  which `Curl_easy` uses that descriptor. This is necessary for the
941  `multi_socket` API.
942 
943  `->conn_cache` points to the connection cache. It keeps track of all
944  connections that are kept after use. The cache has a maximum size.
945 
946  `->closure_handle` is described in the 'connectdata' section.
947 
948  The libcurl source code generally use the name 'multi' for the variable that
949  points to the `Curl_multi` struct.
950 
951 ## Curl_handler
952 
953  Each unique protocol that is supported by libcurl needs to provide at least
954  one `Curl_handler` struct. It defines what the protocol is called and what
955  functions the main code should call to deal with protocol specific issues.
956  In general, there's a source file named [protocol].c in which there's a
957  "struct `Curl_handler` `Curl_handler_[protocol]`" declared. In url.c there's
958  then the main array with all individual `Curl_handler` structs pointed to
959  from a single array which is scanned through when a URL is given to libcurl
960  to work with.
961 
962  `->scheme` is the URL scheme name, usually spelled out in uppercase. That's
963  "HTTP" or "FTP" etc. SSL versions of the protocol need their own
964  `Curl_handler` setup so HTTPS separate from HTTP.
965 
966  `->setup_connection` is called to allow the protocol code to allocate
967  protocol specific data that then gets associated with that `Curl_easy` for
968  the rest of this transfer. It gets freed again at the end of the transfer.
969  It will be called before the 'connectdata' for the transfer has been
970  selected/created. Most protocols will allocate its private
971  'struct [PROTOCOL]' here and assign `Curl_easy->req.protop` to point to it.
972 
973  `->connect_it` allows a protocol to do some specific actions after the TCP
974  connect is done, that can still be considered part of the connection phase.
975 
976  Some protocols will alter the `connectdata->recv[]` and
977  `connectdata->send[]` function pointers in this function.
978 
979  `->connecting` is similarly a function that keeps getting called as long as
980  the protocol considers itself still in the connecting phase.
981 
982  `->do_it` is the function called to issue the transfer request. What we call
983  the DO action internally. If the DO is not enough and things need to be kept
984  getting done for the entire DO sequence to complete, `->doing` is then
985  usually also provided. Each protocol that needs to do multiple commands or
986  similar for do/doing need to implement their own state machines (see SCP,
987  SFTP, FTP). Some protocols (only FTP and only due to historical reasons) has
988  a separate piece of the DO state called `DO_MORE`.
989 
990  `->doing` keeps getting called while issuing the transfer request command(s)
991 
992  `->done` gets called when the transfer is complete and DONE. That's after the
993  main data has been transferred.
994 
995  `->do_more` gets called during the `DO_MORE` state. The FTP protocol uses
996  this state when setting up the second connection.
997 
998  ->`proto_getsock`
999  ->`doing_getsock`
1000  ->`domore_getsock`
1001  ->`perform_getsock`
1002  Functions that return socket information. Which socket(s) to wait for which
1003  action(s) during the particular multi state.
1004 
1005  ->disconnect is called immediately before the TCP connection is shutdown.
1006 
1007  ->readwrite gets called during transfer to allow the protocol to do extra
1008  reads/writes
1009 
1010  ->defport is the default report TCP or UDP port this protocol uses
1011 
1012  ->protocol is one or more bits in the `CURLPROTO_*` set. The SSL versions
1013  have their "base" protocol set and then the SSL variation. Like
1014  "HTTP|HTTPS".
1015 
1016  ->flags is a bitmask with additional information about the protocol that will
1017  make it get treated differently by the generic engine:
1018 
1019  - `PROTOPT_SSL` - will make it connect and negotiate SSL
1020 
1021  - `PROTOPT_DUAL` - this protocol uses two connections
1022 
1023  - `PROTOPT_CLOSEACTION` - this protocol has actions to do before closing the
1024  connection. This flag is no longer used by code, yet still set for a bunch
1025  of protocol handlers.
1026 
1027  - `PROTOPT_DIRLOCK` - "direction lock". The SSH protocols set this bit to
1028  limit which "direction" of socket actions that the main engine will
1029  concern itself with.
1030 
1031  - `PROTOPT_NONETWORK` - a protocol that doesn't use network (read file:)
1032 
1033  - `PROTOPT_NEEDSPWD` - this protocol needs a password and will use a default
1034  one unless one is provided
1035 
1036  - `PROTOPT_NOURLQUERY` - this protocol can't handle a query part on the URL
1037  (?foo=bar)
1038 
1039 ## conncache
1040 
1041  Is a hash table with connections for later re-use. Each `Curl_easy` has a
1042  pointer to its connection cache. Each multi handle sets up a connection
1043  cache that all added `Curl_easy`s share by default.
1044 
1045 ## Curl_share
1046 
1047  The libcurl share API allocates a `Curl_share` struct, exposed to the
1048  external API as "CURLSH *".
1049 
1050  The idea is that the struct can have a set of its own versions of caches and
1051  pools and then by providing this struct in the `CURLOPT_SHARE` option, those
1052  specific `Curl_easy`s will use the caches/pools that this share handle
1053  holds.
1054 
1055  Then individual `Curl_easy` structs can be made to share specific things
1056  that they otherwise wouldn't, such as cookies.
1057 
1058  The `Curl_share` struct can currently hold cookies, DNS cache and the SSL
1059  session cache.
1060 
1061 ## CookieInfo
1062 
1063  This is the main cookie struct. It holds all known cookies and related
1064  information. Each `Curl_easy` has its own private CookieInfo even when
1065  they are added to a multi handle. They can be made to share cookies by using
1066  the share API.
